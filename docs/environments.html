Flight Delay v2 — Environment Strategy

Environments: Dev · UAT · Production Branching: Git Flow Last updated: 2026-04-02 Internal use only

1. Environment Summary

The platform operates three environments. Development runs locally via Docker Compose. UAT is a lightweight Azure environment used for integration testing before every production release. Production is the live customer-facing deployment. All three use the same Docker image and Node.js application — the environment is controlled by NODE_ENV and Key Vault secrets.

2. Git Flow & Branching

The branching model follows a simplified Git Flow: feature branches merge to develop, which auto-deploys to UAT. A manual PR promotes develop to main, which auto-deploys to production.

Hotfix Process

For urgent production fixes, branch from main, fix, merge to main (deploys to prod), then also merge to develop to keep branches in sync.

Never merge directly to main without UAT sign-off — except for genuine hotfixes where the risk of the bug outweighs the testing gap.

6. Safety Controls

UAT must never process real customer data, issue live payments, or trigger live policy lookups. The following controls are enforced in code and infrastructure.

Risk Control Enforced by
Real customer registers in UAT NODE_ENV=uat forces stub policy validator — returns mock data regardless of tenant config Code: policyValidator.js
Real payment sent to customer Modulr stub mode forced in UAT — no live Modulr credentials exist in UAT Key Vault Code: modulr.js + Key Vault secrets
Tester confused about environment Yellow "⚠️ UAT ENVIRONMENT — NOT FOR REAL REGISTRATIONS" banner visible in both admin SPA and customer SPA customer/index.html + admin/index.html
Real email sent to customer DEV_EMAIL_OVERRIDE set in UAT Key Vault — all outbound email redirected to internal test inbox Key Vault secret + notificationService.js
UAT data bleeds into production Separate database server, separate Key Vault, separate resource group — zero shared state Azure infrastructure isolation

How stub enforcement works

Both policyValidator.js and modulr.js check NODE_ENV before making any live API call. If the value is uat or development, the stub path is always taken — even if the tenant's database record has policy_api_mode = 'live' or modulr_mode = 'live'. This prevents a misconfigured tenant from accidentally triggering real integrations in UAT.

UAT banner

The server exposes NODE_ENV via GET /api/tenant-config as the env field. Both SPAs read this field on load and show a fixed yellow banner when env === 'uat'.

7. Reserved Slug Blocklist

Tenant slugs form subdomains (slug.delayedpaid.co.uk). Certain slugs are reserved to prevent conflicts with infrastructure subdomains and UAT. The blocklist is enforced in POST /api/admin/tenants (create) and PUT /api/admin/tenants/:id (rename). 

const RESERVED_SLUGS = [
  'uat', 'www', 'api', 'admin', 'health',
  'mail', 'smtp', 'staging', 'dev', 'test'
];

Attempting to create or rename a tenant to one of these slugs returns HTTP 400.

8. UAT → Production Promotion

Production is updated by merging develop into main. GitHub Actions detects the push and deploys automatically. Target turnaround from merge to live: ~3 minutes.

  1. UAT testing complete — internal sign-off obtained
  2. Open a pull request in GitHub: develop → main
  3. Review the diff — confirm only intended changes are included
  4. Merge the PR (no second approver required at current team size)
  5. GitHub Actions deploy-prod.yml triggers: builds image → pushes to fdv2acr → updates fdv2-app
  6. Production is live. Verify at https://admin.delayedpaid.co.uk
The UAT workflow tags images as develop-<sha> and develop-latest. The production workflow builds a fresh image tagged <sha> and latest. Both pipelines authenticate to Azure via OIDC (workload identity federation) — no long-lived credentials in GitHub secrets.
