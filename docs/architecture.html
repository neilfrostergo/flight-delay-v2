Flight Delay v2 — Architecture Reference

Region: UK South Resource Group: fdv2-prod-rg Runtime: Node.js 20 / Express 5 / PostgreSQL 16 Last updated: 2026-04-01 Internal use only

1. Platform Overview

Flight Delay v2 is a multi-tenant SaaS platform that automates insurance payouts when a covered flight is delayed or cancelled beyond a configurable threshold. Each insurer brand (tenant) gets its own subdomain, white-labeled customer portal, policy validation configuration, and Modulr Faster Payments account. A single Express application serves all tenants; tenant context is resolved from the Host header at every request.

Key design principles

  • No public IP on the application — Container Apps accepts traffic only from Front Door via Private Link.
  • All secrets stored in Azure Key Vault; accessed by the container via Managed Identity — no secrets in environment variables in production.
  • All inter-service traffic stays inside the VNet (10.100.0.0/16) or over Private Endpoints.
  • Monetary values are always integer pence — no floating point.
  • Bank details (sort code, account number) are AES-256-GCM encrypted at rest; decrypted only inside delayProcessor.js immediately before a Modulr payout.
  • OAG flight subscriptions are global — shared across tenants watching the same flight.

7. Application Architecture

The application is a single Node.js 20 / Express 5 process. All three frontends (customer SPA, admin SPA, landing page) are served as static files from the same container. Background event processing runs as a startup service in the same process.

Multi-tenancy resolution

Every request passes through resolveTenant.js. It strips the subdomain from the Host header, looks up the tenant slug in the DB, and attaches the full tenant row to req.tenant. A 30-second in-memory Map cache prevents DB hits on every request. Admin routes call adminTenantScope(req) which returns null for superadmin (no WHERE filter) or tenant_id for tenant admins — all scoped queries use this to enforce row-level isolation.

Payout engine — delayProcessor.js

When a flight event arrives (cancellation or delay beyond the tenant's threshold), the processor:

  1. Finds all active flight_registrations linked to the subscription (may span multiple tenants)
  2. Checks idempotency — skips if a paid or processing payment already exists
  3. Decrypts AES-256-GCM bank details (sort code + account number) — only decrypted at this point
  4. Inserts a payments row in processing state before calling Modulr
  5. Calls modulr.sendPayment() via NAT Gateway static IP
  6. On success: wraps UPDATE payments, UPDATE flight_registrations, and conditional UPDATE registrations in a DB transaction
  7. Fires sendPayoutNotification() (fire-and-forget email via ACS SMTP)

9. Security Controls

ControlMechanismStatus
No public application IP Container Apps uses external ingress accepted only from Front Door Private Link. No public load balancer IP on the ACA environment. Active
Web Application Firewall Azure Front Door Premium — OWASP 2.1 ruleset, Prevention mode. Blocks SQLi, XSS, and other OWASP Top 10 attacks at the edge. Active
DDoS protection Front Door Premium includes DDoS Standard. Traffic absorbs at Microsoft's global network before reaching origin. Active
Secrets management Azure Key Vault (private endpoint, no public access). Container accesses secrets via Managed Identity — no credentials in environment variables or image layers. Active
Database private access PostgreSQL Flexible Server VNet-injected into fdv2-data-subnet. No public endpoint. DNS A-record resolves to 10.100.8.5 only inside VNet. Active
TLS everywhere Front Door terminates external TLS (auto-renewed cert). Internal VNet traffic does not require re-encryption. Event Hub AMQP over TLS via private endpoint. Active
Static egress IP NAT Gateway routes all container outbound traffic through 20.50.124.92. Enables IP whitelisting at Modulr, OAG, and Ergo Connect. Active
Bank detail encryption Sort codes and account numbers encrypted with AES-256-GCM before database insert (encryption.js). Never returned in API responses. Only decrypted inside delayProcessor.js immediately before Modulr payment call. Active
JWT authentication Admin routes require Bearer JWT signed with JWT_SECRET (8h expiry). Payload: { sub, username, role, tenant_id }. Superadmin has tenant_id: null. Active
Password hashing bcryptjs with 12 rounds. Pure JS implementation used for ARM64 Container Apps host compatibility. Active
Rate limiting Three in-process Express rate limiter pools: registrationLimiter, validateLimiter, loginLimiter. Keyed by IP address. Active
CORS Admin API restricted to ADMIN_CORS_ORIGIN. Public customer API is open (serves white-labeled tenant SPAs from any subdomain). Active
CSP headers Helmet.js enforces Content Security Policy. upgrade-insecure-requests enabled in production. Script/style sources restricted. Active
Container image supply chain Images built in GitHub Actions and pushed to private ACR. No image pulled from public registry in production. ACR pull via Managed Identity. Active
CI/CD credential security OIDC Workload Identity Federation — GitHub Actions authenticates to Azure with a short-lived OIDC token. No long-lived client secret stored anywhere. Active
Audit logging All /api/* requests written to request_log table (tenant, status, duration). Admin sensitive actions (key reveals, simulations, status patches) written to audit_log. 90-day retention in Log Analytics. Active
Tenant data isolation All admin queries scoped by adminTenantScope(req). Tenant admin cannot read other tenants' data. Superadmin (tenant_id = NULL) has unrestricted access. Active
Event Hub public access Event Hub Namespace has public access disabled. OAG ingests events via private endpoint 10.100.9.5 inside VNet. Active
Alert rules Azure Monitor alert rules on container restart count and 5xx error rate. Notifications to configured email / PagerDuty. Active

12. Troubleshooting

Application cannot connect to PostgreSQL

Symptoms

Server exits on startup with connection refused or ECONNREFUSED 10.100.8.5:5432.

Checks

  1. Verify DATABASE_URL secret in Key Vault is correct.
  2. Check that Container Apps environment is VNet-integrated with fdv2-aca-subnet.
  3. Verify PostgreSQL DNS: inside the container, nslookup fdv2-postgres.postgres.database.azure.com should resolve to 10.100.8.5.
  4. Check that the PostgreSQL private DNS zone postgres.database.azure.com is linked to fdv2-vnet.
  5. Check PostgreSQL firewall — should have no public rules and only VNet access.

Front Door returns 502 Bad Gateway

Symptoms

Browser receives 502. Front Door health probe shows origin unhealthy.

Checks

  1. Check Container App is running: az containerapp show --name fdv2-app --resource-group fdv2-prod-rg --query properties.runningStatus
  2. Check health endpoint directly from a container exec: curl http://localhost:3000/api/health
  3. Verify Private Link connection is Approved in Front Door origin settings.
  4. Check Container Apps environment uses External ingress type (required for Private Link from Front Door).
  5. Review container logs for startup errors.

Event Hub consumer not processing events

Symptoms

Flights trigger no payouts. flight_events rows accumulate with processed_at = NULL.

Checks

  1. Check logs for [eventHubConsumer] lines. Should show partition ownership on startup.
  2. Verify EVENT_HUB_CONNECTION_STRING and EVENT_HUB_STORAGE_CONNECTION_STRING secrets are set and valid.
  3. Verify Event Hub private endpoint is approved: az network private-endpoint-connection list --name fdv2-eventhub --type Microsoft.EventHub/namespaces
  4. Check checkpoint storage container event-hub-checkpoints exists in fdv2prodsa.
  5. Confirm NODE_ENV=production — development mode uses DB poller, not Event Hub consumer.

Tenant not resolved (returns landing page instead of customer SPA)

Symptoms

Visiting ergo.delayedpaid.co.uk shows the generic landing page, not the tenant's SPA.

Checks

  1. Verify DNS: dig ergo.delayedpaid.co.uk — should CNAME to Front Door endpoint.
  2. Check BASE_DOMAIN env var is set to delayedpaid.co.uk.
  3. Verify tenant row exists in DB: SELECT slug, is_active FROM tenants WHERE slug = 'ergo';
  4. If recently updated, tenant cache TTL is 30s — wait or call invalidateTenantCache(slug) by restarting the container.

CI/CD deployment fails at Azure login step

Symptoms

GitHub Actions step Log in to Azure fails with authentication error.

Checks

  1. Verify AZURE_CLIENT_ID, AZURE_TENANT_ID, AZURE_SUBSCRIPTION_ID secrets are set in the GitHub repository settings.
  2. Verify the GitHub Actions workflow has permissions: id-token: write.
  3. Check the OIDC federated credential on the app registration is configured for the correct repo and branch (repo:neilfrostergo/flight-delay-v2:ref:refs/heads/main).
  4. Verify the managed identity / service principal has AcrPush on fdv2acr and ContainerApp Contributor on fdv2-prod-rg.

Payout fails — payment stuck in processing state

Symptoms

Payment row has status = 'processing' and does not update. No paid email received by customer.

Checks

  1. Check [delayProcessor] log lines for the flight registration ID.
  2. Verify outbound traffic is routing through NAT Gateway: Modulr should see requests from 20.50.124.92.
  3. Verify Modulr API key for the tenant is correctly encrypted and stored in DB (modulr_api_key_enc column on tenants).
  4. Check payments.failure_reason column for any error message written by Modulr.
  5. Use the admin /api/admin/payments retry endpoint to re-attempt.
