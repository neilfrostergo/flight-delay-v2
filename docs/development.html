Flight Delay v2 — Development Workflow

Branching: Git Flow Local dev: Docker Compose Last updated: 2026-04-02 Internal use only

1. Workflow Overview

All changes follow the same path: develop locally → test in browser → raise a Pull Request → merge to develop (auto-deploys to UAT) → test in UAT → merge to main (auto-deploys to production). Nothing reaches production without passing through UAT first.

2. Local Setup

One-time setup to get the development environment running on your machine.

Prerequisites

ToolPurpose
Docker DesktopRuns the app and database locally
GitVersion control
GitHub CLI (gh)Creating pull requests from the terminal
Azure CLI (az)Managing UAT and production infrastructure

First-time setup

# 1. Copy the environment file
cp server/.env.example server/.env

# 2. Edit server/.env — fill in DATABASE_URL, JWT_SECRET, ENCRYPTION_KEY
#    Generate secrets:
node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"  # JWT_SECRET
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"  # ENCRYPTION_KEY

# 3. Start the database
docker compose up db -d

# 4. Run database migrations
docker compose run --rm app npm run migrate

# 5. Seed the superadmin user (set ADMIN_SEED_PASSWORD in .env first)
docker compose run --rm app npm run seed:admin

# 6. Start the full app
docker compose up

The app will be available at http://localhost:3000 (customer portal) and http://localhost:3000/admin (admin panel).

In development, pass the X-Tenant-Slug: demo header to target the demo tenant — there are no real subdomains locally. Use the ModHeader browser extension to set this header automatically.

Wipe and reset

docker compose down -v   # destroys the database volume
docker compose up db -d
docker compose run --rm app npm run migrate
docker compose run --rm app npm run seed:admin
docker compose up

3. Making Changes

Every change — no matter how small — starts on a feature/* branch. Never commit directly to develop or main.

Start a new feature branch

Always branch from develop (not main):

git checkout develop
git pull origin develop
git checkout -b feature/my-change-description

Branch naming

TypePatternExample
New featurefeature/<description>feature/ergo-privacy-link
Bug fixfix/<description>fix/email-override-not-applying
Hotfix (prod)hotfix/<description>hotfix/payment-retry-crash
Always run git pull origin develop before creating a new branch to make sure you're working from the latest code.

4. Testing Locally

Test every change in Docker Compose before raising a PR. This is the fastest feedback loop — no waiting for deployments.

docker compose up

If you've changed server code, Docker Compose will hot-reload automatically. If you've changed the Dockerfile or package.json, rebuild first:

docker compose up --build

What to check

Change typeWhat to verify
Customer SPA changes http://localhost:3000 with X-Tenant-Slug: demo header
Admin SPA changes http://localhost:3000/admin — log in as superadmin
API / server changes Run through the affected flow in the browser or use curl
Database migrations docker compose run --rm app npm run migrate — confirm it runs without error
Email changes Check Mailtrap inbox — all local emails are caught there
Happy with it locally? You're ready to raise a PR.

5. Raising a Pull Request

When local testing is done, commit your changes and raise a PR to develop.

Commit and push

git add <files>
git commit -m "feat: describe what changed"
git push origin feature/my-change-description

Create the PR

gh pr create \
  --base develop \
  --head feature/my-change-description \
  --title "Short description of change" \
  --body "What changed and why"

This outputs a GitHub URL — open it to review the diff. When you're satisfied, click Merge on GitHub.

Merging to develop triggers an automatic UAT deployment (~3 minutes). You don't need to do anything else.

Commit message conventions

PrefixUse forExample
feat:New feature or changefeat: add privacy policy link
fix:Bug fixfix: trust proxy for rate limiting
chore:Config, deps, toolingchore: update node dependencies
docs:Documentation onlydocs: add development workflow guide

6. UAT Deployment

UAT deploys automatically when a PR is merged to develop. GitHub Actions builds a new Docker image, pushes it to fdv2acr, and updates fdv2-uat-app. The whole process takes about 3 minutes.

Accessing UAT

MethodURL
Via domain (once live) https://demo.uat.delayedpaid.co.uk
Direct ACA URL (always works) https://fdv2-uat-app.blackdesert-d8445a44.uksouth.azurecontainerapps.io with X-Tenant-Slug: demo header
Admin panel Append /admin to either URL above

UAT smoke test checklist

CheckExpected
Yellow UAT banner visible"⚠️ UAT ENVIRONMENT — NOT FOR REAL REGISTRATIONS" at top of page
Customer registration flowAll 6 steps complete without error
Policy validationStub returns mock data — no real Ergo Connect call
Confirmation emailLands in Mailtrap inbox, not a real customer address
Admin panel loginSuperadmin can log in and see the registration
Your specific changeVerify whatever you changed works as expected
UAT uses real SMTP (Mailtrap) but DEV_EMAIL_OVERRIDE redirects all emails to the internal test inbox. No real customer will ever receive a UAT email.

Check deployment status

az containerapp revision list \
  --name fdv2-uat-app \
  --resource-group fdv2-uat-rg \
  --query "[0].{revision:name,status:properties.healthState}" -o table

7. Production Deployment

Production is updated by merging develop into main. This is a deliberate manual step — UAT sign-off required before promoting.

  1. UAT testing complete and you're satisfied with the change
  2. Raise a PR on GitHub: develop → main 
    gh pr create \
  --base main \
  --head develop \
  --title "Release: describe what's going to production" \
  --body "UAT sign-off complete. Changes: ..."
  3. Review the diff on GitHub — confirm only intended changes are included
  4. Merge the PR
  5. GitHub Actions deploys to production automatically (~3 minutes)
  6. Verify at https://admin.delayedpaid.co.uk
Never merge directly to main without UAT sign-off — except genuine hotfixes where the risk of the live bug outweighs the testing gap.

Check deployment status

az containerapp revision list \
  --name fdv2-app \
  --resource-group fdv2-prod-rg \
  --query "[0].{revision:name,status:properties.healthState}" -o table

8. Hotfixes

For urgent bugs in production that can't wait for a full UAT cycle, branch directly from main, fix, and merge back to main. Then immediately backport to develop to keep branches in sync. 

# 1. Branch from main
git checkout main
git pull origin main
git checkout -b hotfix/describe-the-fix

# 2. Make the fix, commit
git add <files>
git commit -m "fix: describe the fix"
git push origin hotfix/describe-the-fix

# 3. PR hotfix → main (deploys to production)
gh pr create --base main --head hotfix/describe-the-fix --title "Hotfix: ..."

# 4. After merging to main, ALSO merge to develop
gh pr create --base develop --head hotfix/describe-the-fix --title "Backport: ..."
Always backport hotfixes to develop. If you don't, the fix will be overwritten the next time develop is promoted to main.

9. Branch Conventions

BranchPurposeDeploys toWho merges
main Production-ready code Production (auto) You — after UAT sign-off
develop Integration branch — latest tested changes UAT (auto) You — after local testing
feature/* Individual changes in progress Nowhere (local only) You — via PR to develop
hotfix/* Urgent production fixes Production (via main) You — via PR to main + backport to develop
Rule of thumb: main is always deployable. develop is always UAT-tested. Feature branches are works in progress.

10. Useful Commands

Local development

docker compose up              # Start app + database
docker compose up --build      # Rebuild image first (after Dockerfile/package.json changes)
docker compose up db -d        # Start database only (background)
docker compose down            # Stop everything
docker compose down -v         # Stop and wipe database

Database

docker compose run --rm app npm run migrate     # Run migrations
docker compose run --rm app npm run seed:admin  # Seed superadmin user

Git

git checkout develop && git pull origin develop          # Get latest develop
git checkout -b feature/my-change                        # New feature branch
git add <files> && git commit -m "feat: ..."            # Commit
git push origin feature/my-change                       # Push branch

Pull requests

# Feature → UAT
gh pr create --base develop --head feature/my-change --title "..." --body "..."

# UAT → Production
gh pr create --base main --head develop --title "Release: ..." --body "..."

# View open PRs
gh pr list

UAT operations

# View live logs
az containerapp logs show --name fdv2-uat-app --resource-group fdv2-uat-rg --tail 50

# Check current revision
az containerapp revision list --name fdv2-uat-app --resource-group fdv2-uat-rg --query "[0].name" -o tsv

# Force new revision (picks up Key Vault secret changes)
az containerapp update --name fdv2-uat-app --resource-group fdv2-uat-rg --set-env-vars NODE_ENV=uat

Production operations

# View live logs
az containerapp logs show --name fdv2-app --resource-group fdv2-prod-rg --tail 50

# Check current revision
az containerapp revision list --name fdv2-app --resource-group fdv2-prod-rg --query "[0].name" -o tsv
